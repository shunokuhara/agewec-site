# AGEWEC マルチイヤー構成 設計メモ（v0.1）

毎年（2026, 2027, …）開催する前提での、サイト・データの構成方針を記録する。

## 目的

- 毎年版を継続的に開催する。
- URL は `agewec.com/2026`、`agewec.com/2027` のパス方式（1ドメインで運用が楽）。
- コードは1つに保ち、年ごとに複製しない（バグ修正を1か所で全年に反映するため）。

## 決定事項

### 1. URL はパス方式（年を先頭セグメントにする）

`agewec.com/{year}/...` の形にする。サブドメイン（`2026.agewec.com`）ではなくパスを採用。理由は運用の容易さ（DNS・証明書を年ごとに増やさない）。

注意：単純に `/2026/` 配下へ静的ファイルを置くと、ページの絶対パス（`/styles.css`、`/api/submit`）がずれて404になる。これは「Worker が年を意識する」設計で解決する（下記）。

### 2. 単一コードベース・ページは複製しない

応募フォーム・審査・管理・公開・規約などのページは年に依存しない共通テンプレート。実体は1セットだけ持ち、年ごとに複製しない。年で変わるのはトップページの文言（開催日・会場・テーマ）と締切くらい。

### 3. データは「年ごとに別D1」（推奨・採用）

各年に独立した D1（`agewec_2026`, `agewec_2027`, …）を用意し、Worker が年で接続先を切り替える。

採用理由：

- アーカイブ・削除が年単位で安全（その年のDBをエクスポート凍結／丸ごと削除でき、他年に影響しない）。プライバシーポリシーの年単位の保管・削除方針と整合。
- 個人情報の漏えい範囲が1年分に限定（ブラスト半径が小さい）。
- 追加手間は年ごとに `wrangler d1 create` ＋ バインド1行のみ。

却下した代替案：**1つのD1＋`year`列**。最短で立ち上がるが、全年のPIIが同居し、`WHERE year=...` の事故で別年を巻き込むリスク、漏えい範囲が全年に及ぶ。最短優先のときのみ検討。

## 仕組み（年対応ルーティング）

- **共有アセット（CSS/JS/画像）はルート直下**のまま：`/styles.css`、`/script.js`、`/assets/`。年で変わらないので全年共通。
- **Worker が最初のパス区切りを年として扱う**：
  - `/2026/api/...` → 2026 の D1 を使って API 処理。
  - `/2026/submit/` 等のページ → 年を剥がして共通ページを配信（`env.ASSETS.fetch('/submit/')`）。`/2026/...` はどの静的ファイルにも一致しないため必ず Worker に届く。
- **ページ側は年プレフィックスを付ける**：URL先頭4桁から年を読み（`location.pathname`）、API呼び出しと内部リンクに前置き。`fetch('/api/submit')` → `fetch('/'+Y+'/api/submit')`、`/submit/` → `/${Y}/submit/`。

## 共有 vs 年ごと

| 対象 | 区分 |
|---|---|
| Worker コード（`worker/index.js`） | 共有（全年1つ） |
| ページ（submit / judge / admin / entries / rules / privacy / ai-guidelines） | 共有テンプレート |
| CSS / JS / 画像 | 共有（ルート直下） |
| 採点ルーブリック | 共有（年で変えるなら rubric_version で管理） |
| D1（応募・採点データ） | **年ごとに別** |
| トップページの文言（年・開催日・会場・テーマ） | 年ごと（設定値から差し込む） |
| 締切・受付状態 | 年ごと（各年D1の settings） |

## 新年版の追加手順（runbook）

1. `wrangler d1 create agewec_2027`
2. `wrangler.jsonc` に 2027 用 D1 バインドを追加（Worker が年→D1 を解決できるように）。
3. `wrangler d1 migrations apply agewec_2027 --remote`（スキーマ適用）。
4. その年の judges/admin を登録。
5. トップの年・開催日・会場・締切の設定値を更新。
6. デプロイ（コード共有なので `wrangler deploy` 1回）。

ページもコードも複製なし。バグ修正は1か所で全年に反映。

## データのライフサイクル（年単位）

- 開催中：その年のD1に応募・採点が蓄積。
- 終了後：エクスポートしてバックアップ／凍結。公開作品は `/entries` で継続公開可。
- 保管期間経過後：その年のD1を削除または匿名化（他年に影響なし）。

## 凍結のルール（再掲）

- フォーム項目は応募開始前に凍結。
- 採点ルーブリックは審査開始前に凍結。
- 年をまたぐ変更は additive（列追加）中心。各行に form/rules/privacy/rubric の版を保存済み。

## 実装ステータス

- 現状（v1）：単年構成。トップ・submit・judge・admin・entries・規約類・Worker・D1（単一）・CI まで実装済み、24項目テスト通過。
- 未実装（このメモの対象）：Worker の年ルーティング、ページの年プレフィックス、`wrangler.jsonc` の年ごとD1バインド、トップ文言の設定値化。
